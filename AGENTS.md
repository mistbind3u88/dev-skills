# AGENTS.md

## このリポジトリの目的

特定のプロジェクトや業務領域に依存しない、汎用的な開発ワークフロースキルを管理する。

## スキルの追加・変更に関する制約

### 含めてよいもの

- git / GitHubの操作に関するスキル
- Claude Codeのセットアップや設定に関するスキル
- コードレビュー、リント、フォーマットなど汎用的な開発プロセスのスキル
- 言語やフレームワークに依存しない汎用ツール連携のスキル

### 含めてはいけないもの

- **業務領域固有の知識やロジック** — 特定プロジェクトのドメインモデル、ビジネスルール、社内用語への依存
- **秘匿情報** — APIキー、トークン、認証情報、社内URL、内部システム名
- **特定組織のインフラ情報** — CI/CDパイプラインの詳細設定、デプロイ先のホスト名、内部ネットワーク構成
- **特定リポジトリの構造に依存するスキル** — 特定のディレクトリ構成やファイル命名規則を前提とするもの

### 判断基準

スキルを追加・変更する際は以下を確認すること:

1. **公開リポジトリである** — pushされる内容はすべて公開される前提で書く
2. **汎用性** — git/GitHubを使う任意のプロジェクトで利用できるか
3. **独立性** — 特定のリポジトリをクローンしなくても単体で動作するか
4. **最小権限** — `allowed-tools` はスキルの目的に必要なコマンドとサブコマンドに限定する。包括的な許可 (例: `Bash(npm:*)`) ではなく、実行するサブコマンド単位 (例: `Bash(npm run:*)`, `Bash(npm test:*)`) で指定する。他のスキルに委譲できる操作は自スキルの `allowed-tools` に含めず、`/skill-name` 呼び出しで委譲する。スキル更新時には不要になった許可を必ず削除する。スキル配下に実装しているスクリプトやツールは `allowed-tools` に含めない（`Bash($SKILL_DIR/foo.sh:*)` のようにスキル相対パスで書いても Claude Code が argv[0] を runtime-determined として毎回承認を要求するため機能しない）。該当スクリプトは PATH 上に配置し bare name で呼び出す前提とする
5. **MECE** — スキル間で機能の重複や漏れがないよう設計する。既存スキルと責務が重なる場合は、既存スキルの拡張や呼び分けで対応し、新規スキルの追加は避ける
6. **循環参照禁止** — スキル間の `/skill-name` 呼び出しが循環しないことを確認する。作成・更新時には呼び出し先を辿って循環がないかチェックする
7. **ファイル命名規則** — スキル定義ファイルは `SKILL.md`、エージェント向け指示ファイルは `AGENTS.md` とし、常に大文字で命名する。`AGENTS.md` を配置する場合は `CLAUDE.md` → `AGENTS.md` のシンボリックリンクを必ずセットで作成する

## スキルの分類

### 単機能スキル

単一の責務を持つスキル。MECE の対象であり、スキル間で機能が重複しないように設計する。自身で `allowed-tools` を持ち、具体的な操作を実行する。

例: `commit`、`check`、`push`、`gh-edit`、`mark`

### ワークフロースキル

複数の単機能スキルを決まった順序で呼び出すラッパー。自身では操作を行わず、オーケストレーションのみを担う。`allowed-tools` は空にする。

- ワークフロースキルは MECE の対象外。単機能スキルの組み合わせを定義するものであり、機能の重複ではない
- 単機能スキルがワークフロースキルを呼び出してはならない（循環参照の原因になる）
- いずれかのステップが失敗したらワークフロー全体を停止する

例: `ship`（`/check` → `/commit` → `/push` → `/gh-edit`）
