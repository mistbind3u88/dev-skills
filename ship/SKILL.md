# ship スキル

変更を仕上げて送り出すまでの一連の手順を実行する。

## 手順

以下のスキルを順番に実行する。いずれかが失敗したら停止し、どのステップで止まったかを報告する。

### 1. `/commit`

変更をコミットする。

### 2. `/check`

未チェック項目（lint・build・test・review）を実行する。

### 3. `/push`

チェック済みであることを確認して push する（ステップ2で通過済みのためスキップが期待される）。

### 4. PR の存在を確認する

```bash
gh pr view --json number,title,state --jq '.'
```

PR が存在しない場合は `/gh-edit` で新規作成する。

### 5. `/gh-edit` と `/watch-ci` を並行実行する

push 完了後、以下を並行で実行する:

- `/gh-edit` — PR の概要欄を作成・更新する
- `/watch-ci` — GitHub Actions の CI 実行を監視する

**注意**: PR 作成直後は CI の実行が作成されるまでに時間がかかることが多い。ステップ4で PR を新規作成した場合は、スキル `/watch-ci` を実行する前に 10 秒待つこと。

`/watch-ci` が失敗を報告した場合、その旨をユーザーに伝える。

#### autosquash のみの場合

コード変更がなく autosquash でコミットを整理しただけの場合（ステップ4で PR が既に存在する場合）:

- `/gh-edit` はスキップする（PR 概要欄の更新は不要）
- スキル `/pr-progress` を実行して autosquash の差分コメントを投稿する

## 注意

- 各ステップの詳細は対応するスキルの SKILL.md に従う
- ステップ間でユーザーの判断が必要になった場合（レビュー指摘への対応、コミットメッセージの確認など）、該当スキルの中でユーザーとやり取りしてから次に進む
- `$ARGUMENTS` で `--skip-review` が指定された場合は `/check --skip-review` に渡す
- `$ARGUMENTS` で `--skip-gh-edit` が指定された場合はステップ5の `/gh-edit` をスキップする（`/watch-ci` のみ実行）
- コミットする変更がない場合はステップ1をスキップする
- `/check` の指摘に対応して新たにコミットを作成した場合は、push せずにワークフローを中断する。チェック通過後に追加変更が入ると、CI 上で不整合が検知されるリスクがあるため、再度 `/ship` を実行してもらう
