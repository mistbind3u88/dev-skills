# gh-editスキル

GitHubのPRやIssueを作成・更新する。

**概要欄の読み手は人間である。** データの羅列や機械的な転記ではなく、読み手が背景・目的・判断ポイントを理解できる文章を書くこと。

## 手順

### 1. 引数から対象と操作を特定する

- PR番号、Issue番号、またはURLが指定される
- 未指定の場合は現在のブランチに紐づくPRを対象とする
- 対象が存在しない場合は新規作成する

### 2. テンプレートを探す

PR/Issueの概要欄を作成・更新する前に、テンプレートを探してReadツールで読み込む。

#### テンプレートの優先順位

1. **リポジトリのテンプレート**（優先）: `.github/PULL_REQUEST_TEMPLATE.md` や `.github/ISSUE_TEMPLATE/` 配下のファイルが存在すればそちらを使う
2. **スキル同梱のテンプレート**（フォールバック）: リポジトリにテンプレートがない場合、このスキルの配置ディレクトリにあるテンプレートを使う
   - PR: [AGENTS.md](AGENTS.md) と [pull_request_template.md](pull_request_template.md)
   - Issue: [issue_template.md](issue_template.md)

### 概要欄の構成

PR・Issueを問わず、概要欄の最初の見出しは `# Subject` とし、そのPR/Issueの趣旨を要約する文章を記載する。タイトルの繰り返しではなく、背景・目的・動機が伝わる内容にすること。

### Issue の紐付け

PRが1対1で対応するIssueを解決する場合、概要欄に `Closes #123` を記載する。マージ時にIssueが自動的にCloseされる。直接Closeしない関連Issue/PRは `ref: #456` 等で参照する。

### 3-A. PRの新規作成

ブランチのコミット履歴からタイトルと概要を作成する。PRは常にdraftで作成する。

#### クロスリポジトリ対応時のブランチ名・タイトル統一

同一の対応を別リポジトリに展開する場合は、元のPRとブランチ名・PRタイトルを原則同名にする。関連PRがある場合は `gh pr view` で元のブランチ名・タイトルを確認し、合わせること。

1. `.claude/github/` 配下にPRの概要欄となるドキュメントを作成する

```bash
git log --oneline main..HEAD
git diff --stat main..HEAD
```

2. ドキュメント作成後、手順 4 の執筆方針セルフレビューを経てから手順 5 で PR を作成する。この段階では `gh pr create` を実行しない。

### 3-B. Issueの新規作成

Issueは直接 `gh issue create` するのではなく、まずローカルに設計ドキュメントを書き出し、レビューを経てから投稿する。

1. `.claude/github/` 配下にIssueの概要欄となるドキュメントを作成する
2. ドキュメント作成後、手順 4 の執筆方針セルフレビューを経てから手順 5 で Issue を作成する。この段階では `gh issue create` を実行しない。

#### Sub Issue として作成する場合

以下のように「親 Issue が明確に存在し、その一部を切り出す」作業を Issue 化する場合は、GitHub の Sub Issue 機能で親子関係を設定する。紐付けコメント（`ref: #123` 等）で代用しない。

- 親 Issue から切り出した別 PR で対応する作業
- 親 Issue のスコープ外にした関連対応
- 親 Issue を実装する上での前提・後続タスク

手順:

1. 通常通り `gh issue create` で子 Issue を作成し、発行された Issue 番号を控える
2. 子 Issue の内部 ID（Issue 番号とは別の数値 ID）を取得する

```bash
gh api repos/<OWNER>/<REPO>/issues/<子Issue番号> --jq '.id'
```

3. 親 Issue に Sub Issue として紐付ける

```bash
gh api -X POST repos/<OWNER>/<REPO>/issues/<親Issue番号>/sub_issues \
  -f sub_issue_id=<子Issueの内部ID>
```

**注意**:

- `sub_issue_id` は Issue 番号ではなく、`gh api` で取得できる内部の数値 ID を指定する
- 親 Issue 側にコメントで `ref: #子番号` を追加する対応は取らない（Sub Issue 機能と重複する）
- 子 Issue 側の概要欄には `親 Issue: #<親番号>` のように記載しておき、GitHub UI を経由しなくても関係が追えるようにする

### 3-C. 既存の更新

#### タイトルと概要欄を読み込む

```bash
# PRの場合
gh pr view <番号> --json title,body --jq '.title,.body'

# Issueの場合
gh issue view <番号> --json title,body --jq '.title,.body'
```

**重要**: 既存の内容を必ず確認してから編集すること。白紙から書き直さない。

#### コミット履歴を確認する（PRの場合）

```bash
git log --oneline main..HEAD
git diff --stat main..HEAD
```

#### タイトルと概要欄を更新する

既存の内容をベースに、修正・追記・削除を行う。
変更箇所だけでなく概要欄全体を読み直し、セクション間の重複や矛盾がないか確認する。
タイトルはコミット履歴と概要欄の内容を踏まえて、PR/Issueの現在のスコープを正確に反映しているか見直す。

概要欄のソースファイルがある場合は、ソースファイルを先に更新する。ソースファイルがない場合は `.claude/github/` 配下に新規作成する。

ソースファイルを更新したら、手順 4 の執筆方針セルフレビューを経てから手順 5 で `gh pr edit` / `gh issue edit` を実行する。この段階では `gh ... edit` を実行しない。

### 4. 執筆方針のセルフレビュー

概要欄のソースファイル（`.claude/github/<file>`）を書き出したら、投稿前にサブエージェント（`general-purpose`）を起動して執筆方針への適合を検査する。自分で書いた直後のセルフレビューでは見落としが起きやすいため、必ず独立したエージェントに判定させる。

#### エージェントへの入力

以下をプロンプトにまとめて渡す。

- 対象のソースファイルの絶対パス
- リポジトリ同梱のテンプレート（`.github/PULL_REQUEST_TEMPLATE.md` や `.github/ISSUE_TEMPLATE/`）があればそのパス。なければスキル同梱のテンプレート（`gh-edit/pull_request_template.md` / `gh-edit/issue_template.md`）と `gh-edit/CLAUDE.md`
- ユーザーの記述ルール（存在すれば `~/.claude/rules/issue-pr.md`）
- 対象が PR か Issue か、新規作成か更新かの区別

#### エージェントに判定させる観点

1. **コードの説明に寄りすぎていないか** — 差分の逐語的な説明や実装詳細の羅列になっていないか。Issue は利用者・運用者視点、PR はプロダクトとしての変化と意思決定が主軸になっているか
2. **独自用語・造語を使わず、読み手を意識した表現になっているか** — プロダクト・チームで実際に使われている言葉を使っているか。曖昧な表現（「いい感じに」「適切に」等）を避けて具体的に書かれているか
3. **各見出しに対して記載内容が適切か** — テンプレートの見出し（`# Subject`、`## やったこと`、`## 確認したこと` 等）がリテラルのまま保持されているか。各セクションに想定された内容が書かれており、他セクションとの重複・漏れ・齟齬がないか。空のセクションが残っていないか（該当なしならセクションごと削除されているか）
4. **その他のスタイル規則** — `Closes`/`fixes` 等のキーワードの使い分け、コミットハッシュのコードブロック化禁止、ローカルファイル（`.claude/docs/` 等）への参照禁止など、`gh-edit/CLAUDE.md` と `~/.claude/rules/issue-pr.md` に記載の規則

#### エージェントの出力形式

観点ごとに `OK` / `要修正` を示し、`要修正` の項目は該当箇所（見出し名や抜粋）と修正方針を添えて報告させる。観点を横断する自由記述の所感は不要。

#### 結果の取り扱い

- `要修正` の指摘があればソースファイルを修正し、再度サブエージェントで検査する。最終的に全観点 `OK` になるまで手順 5 に進まない
- 指摘内容に対応しない判断をする場合はユーザーに確認する

### 5. 執筆方針レビュー通過後に投稿する

セルフレビューを通過したソースファイルで、対象に応じたコマンドを実行する。

```bash
# PR 新規作成
gh pr create --draft --title "<タイトル>" --body-file .claude/github/<ドキュメントファイル>

# Issue 新規作成（事前にユーザーの承認を得る）
gh issue create --title "<タイトル>" --body-file .claude/github/<ドキュメントファイル>

# 既存 PR の更新
gh pr edit <番号> --title "<タイトル>" --body-file <ソースファイル>

# 既存 Issue の更新
gh issue edit <番号> --title "<タイトル>" --body-file <ソースファイル>
```

## 注意

- **概要欄は人間が読むものである。** 調査データや分析結果をそのまま貼るのではなく、読み手が順を追って理解できるように構成・要約すること
- コミットハッシュはコードブロック（`` ` ``）で囲まない。GitHub UI 上でコミットへのリンクとして自動認識させるため
- PRは常にdraftで作成する
- 既存の概要欄が空でない場合、必ず既存の内容を読み込んでから更新する
- ユーザーが明示的に全面書き換えを指示しない限り、既存の構造を維持する
- **概要欄を作成・更新する前に、必ずテンプレートを探して読み込むこと**（手順2参照）
- 既存の PR/Issue を更新する際は、会話中に内容を把握していても必ず `gh pr view` / `gh issue view` で最新の内容を読み込む。ユーザーが GitHub UI 上で直接編集している可能性があり、読み込みを省略すると変更を上書きしてしまう
- **「確認したこと」セクションにはmerge前に確認すべき項目を記載する。** merge後にしか確認できない事項（デプロイ後の動作確認など）は含めない。CI等で自動確認される事項も記載しない。該当する確認項目がなければセクションごと削除する
- **スコープ外にした関連 Issue/PR がある場合は「やってないこと」セクション等で明記する。** 対応を別途行うと判断した Issue や、意図的にスコープから除外した事項は、読み手が経緯を追えるように PR/Issue の概要欄に記載する
- **概要欄にローカルファイルへの参照を含めない。** `.claude/docs/` 等のローカルドキュメントはGitHub上で読めないため、概要欄から参照しても読み手が内容を確認できない。必要な情報はIssue/PR本文に直接記載するか、リポジトリにコミットされたファイルを参照する
