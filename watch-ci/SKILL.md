# watch-ci スキル

GitHub Actions の CI 実行を監視し、完了を待って結果を報告する。

## 手順

### 1. CI の情報を確認する

#### ドキュメントの確認

リポジトリの `AGENTS.md`、`CLAUDE.md`、`README.md` に CI の検査内容（実行されるジョブ、チェック項目など）が記載されていれば確認し、ステップ4の分析に活用する。

#### ワークフローの存在確認

`.github/workflows/` 配下にワークフローファイル（`*.yml` / `*.yaml`）が存在するか確認する。

存在しない場合は「CI が設定されていません」と報告して終了する。

### 2. 最新の CI 実行を特定する

```bash
gh run list --branch <ブランチ名> --limit 5 --json databaseId,status,conclusion,workflowName,headSha,createdAt
```

- 現在の HEAD コミットに一致する実行を探す
- 一致する実行がない場合は 10 秒待ってからリトライする（最大 3 回）
- リトライしても見つからない場合は「CI がトリガーされませんでした」と報告して終了する

### 3. CI の完了を監視する

```bash
gh run watch <run-id>
```

複数のワークフローが実行されている場合は、それぞれを監視する。

### 4. 結果を報告する

- 全ワークフロー成功: 成功した旨を報告する
- いずれか失敗: 以下の手順で分析する

#### 失敗時の分析

1. 失敗したワークフローのログを取得する

```bash
gh run view <run-id> --log-failed
```

2. ログからエラーの原因を分類する

| 分類 | 例 | 対応 |
|------|-----|------|
| コード起因 | テスト失敗、lint エラー、ビルドエラー、型エラー | 必要な修正箇所を洗い出してユーザーに報告する |
| 環境起因（一時的な不調の可能性） | DB 接続失敗、ネットワークタイムアウト、レートリミット、外部サービスのエラー、flaky test | 一時的な不調の可能性がある旨を伝え、リトライを提案する |
| 設定起因 | シークレット未設定、権限不足、ランナーのバージョン不一致 | CI 設定の修正が必要な旨を報告する |

3. コード起因の場合は、ログから具体的な修正箇所（ファイル名・行番号・エラー内容）を抽出して報告する

## 注意

- このスキルは監視・分析・報告を行う。コードの修正は行わない
