# codex review スキル

変更差分を codex CLI でレビューに出す。

## 手順

1. git の状態を確認し、レビュー対象の変更範囲を特定する

`$ARGUMENTS` が指定されていれば base ref として使う。未指定なら main を使う。
`git status -s`、`git log`、`git diff --stat` をそれぞれ個別に実行し、未コミットの変更とコミット済みの差分を把握する。

2. 差分の内容からレビュータイトルと観点別プロンプトを作成する

- **タイトル**: PR タイトル相当の簡潔な説明 (conventional commits 形式)。全観点で共通のタイトルを使う。
- **共通コンテキスト**: 各プロンプトに共通で含める情報
  - **コミット範囲**: コミット済みの差分がある場合、BASE_REF と HEAD のハッシュ値を必ず明記する (例: `レビュー対象: abc1234..def5678`)。未コミットの変更がある場合はその旨も記載する
  - 変更概要 (何を、なぜ変更したか)
  - 主な変更点 (箇条書き)
  - 設計判断 (トレードオフや選択の根拠)
  - **関連 PR/Issue**: 対応の背景となる他リポジトリの PR や Issue がある場合、URL を含めて記載する。codex がコンテキストを把握しやすくなる
  - **既知の指摘事項**: 以前のレビューで指摘され、対応不要と判断済みの事項があれば、指摘内容と対応不要の理由を記載する。codex が同じ指摘を繰り返さないようにする

3. codex review を実行する

デフォルトでは観点を絞らず 1 回のレビューを実行する。

```bash
codex review --title "<タイトル>" "<共通コンテキスト + レビュー指示>"
```

### オプション: 観点別レビュー

`$ARGUMENTS` や口頭指示で観点を指定された場合、観点ごとに個別の `codex review` を並列実行する。

| 観点                                 | レビュー指示の要点                                                                     |
| ------------------------------------ | -------------------------------------------------------------------------------------- |
| **設計・実装の妥当性**               | 責務分離、命名、インターフェース設計、エッジケースの網羅、テストの妥当性               |
| **耐障害性**                         | ロールバック可否、部分失敗時の挙動、冪等性、タイムアウト・リトライ、データ不整合の防止 |
| **セキュリティ・データ整合性**       | インジェクション、認可漏れ、入力バリデーション、競合状態、機密情報の露出               |

観点別実行時はタイトルに観点名をサフィックスとして付ける。

```bash
codex review --title "<タイトル> [設計・実装]" "<共通コンテキスト + 設計・実装の観点指示>"
```

### codex review の引数仕様

```
codex review [OPTIONS] [PROMPT]
```

- `--title <TITLE>`: レビュータイトル (必須)
- `-c <key>=<value>`: `~/.codex/config.toml` の設定を一時的に上書き（`model`, `model_reasoning_effort` 等）
- `--uncommitted`: ステージ済み・未ステージ・未追跡の変更をレビュー (**`[PROMPT]` と併用不可**)
- `--base <BRANCH>`: 指定ブランチとの差分をレビュー (**`[PROMPT]` と併用不可**)
- `--commit <SHA>`: 特定コミットの変更をレビュー
- `[PROMPT]`: レビュー指示 (自由記述)。`-` で stdin から読み込み

### モデル設定

review 用のモデル設定は `$SKILL_DIR/config.toml` で管理する。通常の codex 利用（`~/.codex/config.toml`）とは独立した設定。

実行時に config.toml の各キーを `-c` オプションとして渡す:

```bash
# config.toml の内容を -c オプションに変換して渡す
codex review \
  -c model='"..."' \
  -c model_reasoning_effort='"..."' \
  --title "..." "..."
```

**制約**: `--uncommitted`、`--base` はいずれも `[PROMPT]` と同時に使えない。stdin (`-`) もプロンプト扱いで同様に併用不可。

**推奨**: プロンプト付きでレビューしたい場合は `--uncommitted` や `--base` を付けず、プロンプトのみで実行する。codex が自動的に未コミットの変更も含めてレビュー対象を検出する。

4. バックグラウンド実行中にモデル設定を確認する

review をバックグラウンドで実行した後、完了を待つ間に以下を確認する:

- `$SKILL_DIR/config.toml` の現在の設定を読み取る
- `codex --help` や `codex review --help` で利用可能なオプションを確認し、より適切なモデルや推論設定がないか調べる
- 変更が有効な場合はユーザーに `$SKILL_DIR/config.toml` の更新を提案する（次回以降の review に反映される）

5. codex review の結果を確認し、フィードバックを要約する

codex review はバックグラウンドで実行される。完了通知を受け取ったら、出力ファイルを読み込む。

出力は大量のログを含むため、末尾（最後の 200 行程度）から `codex` のレビューコメントを探す。

結果をユーザーに報告する。codex の出力をそのまま転記せず、各指摘を以下の形式で整理すること:

- 対象ファイルと行番号
- 指摘内容の要約（人間にわかりやすい言葉で説明する）
- 深刻度を「要修正」「検討推奨」「軽微」のいずれかで示す（codex 独自の優先度ラベルをそのまま使わない）

報告後、「要修正」の指摘があればユーザーの判断を仰ぐ。

6. レビュー完了後にチェックタグを設置する

全観点のフィードバック報告が完了したら、`/mark review` を実行してチェックタグを設置する。

## 注意

- codex の実行は時間がかかるため、バックグラウンドで実行する
- バックグラウンド実行の完了通知を受けたら、結果の読み込みとフィードバック報告を自動的に行う
